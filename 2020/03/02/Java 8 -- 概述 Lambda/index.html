鉴于是第一次，谨慎些看清楚：

  • @FunctionalInterface 只是标识它是一个函数式接口，并交由编译器检查，类似@Override。

如果一个接口的抽象方法只有一个，那么它就能够成为一个函数式接口。事实上，你可以随便你定义你自己的函数式接口。如果只看函数式接口的应用，以下定义也能够起到和 Predicate 同等的作用：

因此，想用多复杂的 Lambda，就取决于你定义了多少个泛型、参数以及复杂的返回类型。但是，在大多数情况下建议优先使用标准的函数接口，新的接口意味着更多的学习成本，更低的可读性以及更高的维护成本（你爽了团队懵了）。标准的函数接口有六类：

  • UnaryOperator<T>，一元操作。 T apply(T t)，操作T，并得到处理后的T。
  • BinaryOperator<T,T>，二元操作。T apply(T t1, T t2)，操作T1和T2，并得到处理后的T。
  • Predicate<T>，谓词。boolean test(T t)，校验t是否匹配。
  • Function<T,R>，函数，也可以理解成映射。R apply(T t) T 转化成 R
  • Supplier<T>，生产者。 T get()。
  • Consumer<T>，消费者。 void accept(T t)。

除此以外，他们还有基于 int, long, double 的基本类型变体，以及更多操作数的变体。这里就不赘述了，查看java.util.function就好。

Lambda 表达式的结构

回到完整的 Lambda 表达：Predicate<Apple> predicate = (Apple apple) -> "red".equals(apple.getColor())

  • (Apple apple) Lambda 参数。Lambda 参数可以指定类型，也可以借助编译器的类型判断省略类型。如 apple -> "red".equals(apple.getColor())，没有哪个好哪个坏，规范的问题是由团队决定的。我个人倾向后者。
  • -> 箭头，箭头就是用来分开参数列表和主体（提前一秒的剧透）。

变量作用域

当函数可以如变量一般传递时，我们就不得不关注他的作用域了，为什么？没有函数变量时，函数作用域是{}内按照从上往下的作用，但是当函数可被传递时，这个传递进来的函数就成为了“内部函数”，会把“外部函数”的{}一分为二，对于外部函数，它自己的 {}内从上往下，但“内部函数”呢？先直接摆出现象：

闭包 (是个名词)

“内部函数”的作用域会按其词法顺序形成，同时它的局部变量无法被外部函数访问，通俗地说我们把这样构成的一个函数称为闭包。它保留了一部分外部的副本来访问外部的局部变量，并封闭了自己不让外部访问。因此，

  • 当inner作为一个函数的整体传递到另外一个函数outer的内部时，它可以访问outer的局部变量 outside，但无法访问outside2，即 error1，因为outside2在它下面。
  • outer也无法访问innerWeight，即error4，因为 innnerWeight 是在 inner 内部的。

Effectively Final

对于Java语言，还限制了开放的局部变量无法被修改的性质，outside 就是这个开放的局部变量。开放的变量不能被显式地赋值=(++/–当然也不行)，这种性质我们称之为”effectively final”，没有 final 修饰胜似 final。即 error2 和 error3。可能还是有点抽象，直观地看此时的 inner 它更接近于以下结构：
